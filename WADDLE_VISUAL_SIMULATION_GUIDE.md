# Waddle Visual Simulation Guide

This guide explains how to run and inspect Waddle policies in simulation, what controls are available, why the robot may move on its own, and what `.pt` vs `.onnx` files are.

```bash

```

## 1. Which file to use for visual testing

For MJLab `play`:
- Use a training checkpoint: `model_*.pt`

Example:

```bash
uv run play Mjlab-Velocity-Flat-Waddle \
  --agent trained \
  --checkpoint-file logs/rsl_rl/waddle_velocity/<run_folder>/model_250.pt \
  --num-envs 1 \
  --viewer native
```

For ONNX runtime testing (separate script path):
- Use an `.onnx` policy file.
- Script: `scripts/infer_policy.py` (has explicit keyboard command controls).

## 2. Why the robot can "auto bounce" in simulation

For velocity tasks, the policy does not stand still by default.

It receives sampled velocity commands (`twist`) and tries to follow them. If those commands are non-zero, the robot will attempt locomotion and can look unstable early in training.

Also, this repo's base velocity task includes random push disturbances. For Waddle play mode, this was now disabled in:
- `src/mjlab_microduck/tasks/waddle_velocity_env_cfg.py`

Specifically:
- in play mode, `push_robot` is removed so visual evaluation is cleaner.

## 3. Where to change foot raise height and related limits

### Foot raise height targets
File:
- `src/mjlab_microduck/tasks/waddle_velocity_env_cfg.py`

Keys:
- `cfg.rewards["foot_clearance"].params["target_height"]`
- `cfg.rewards["foot_swing_height"].params["target_height"]`

Current Waddle defaults:
- both set to `0.03`

### Velocity command ranges (how fast it tries to move)
File:
- `src/mjlab_microduck/tasks/waddle_velocity_env_cfg.py`

Keys:
- `command.ranges.lin_vel_x`
- `command.ranges.lin_vel_y`
- `command.ranges.ang_vel_z`

Current Waddle defaults:
- `lin_vel_x = (-0.06, 0.10)`
- `lin_vel_y = (-0.10, 0.10)`
- `ang_vel_z = (-1.0, 1.0)`

### Other important anti-bounce knobs
Files:
- `src/mjlab_microduck/tasks/waddle_velocity_env_cfg.py`
- inherited terms in `src/mjlab_microduck/tasks/microduck_velocity_env_cfg.py`

High-impact terms:
- `joint_pos_action.scale`
- `action_rate_l2` reward weight
- `joint_torques_l2` reward weight
- `air_time`, `foot_slip`, `soft_landing` rewards

## 4. What this velocity model trains for

Task id:
- `Mjlab-Velocity-Flat-Waddle`

The model learns a policy that maps observations to joint actions so the robot:
- tracks commanded linear and angular velocity
- stays upright
- keeps useful gait structure (air-time/swing/clearance rewards)
- avoids slip and aggressive behavior (penalties)

So if command sampling asks for motion, the policy will try to move. In early checkpoints this can look like hopping, bouncing, or noisy stepping before stable gait emerges.

## 5. Should you worry about early bouncing checkpoints?

Usually not at very early checkpoints.

Expected in early training:
- erratic vertical motion
- poor heading control
- frequent falls

When to intervene early:
- extreme motion persists after a few hundred iterations
- landing forces/slip remain very high while reward does not improve
- sim motion is clearly outside actuator realism

Practical approach:
1. Test checkpoints at `model_50.pt`, `model_250.pt`, `model_500.pt`.
2. If still too bouncy by `500-1000`, tune action smoothness / command ranges.
3. Keep one change per run so effects are clear.

## 6. Controls in visual tools

## 6.1 MJLab play viewer controls

Command:

```bash
uv run play Mjlab-Velocity-Flat-Waddle --agent trained --checkpoint-file <path_to_model.pt> --num-envs 1 --viewer native
```

Native viewer keys (from MJLab viewer):
- `Space`: pause/resume
- `Enter`: reset environment
- `-` / `=`: slower/faster playback multiplier
- `,` / `.`: previous/next environment index (if `num_envs > 1`)
- `R`: toggle debug visualization
- `P`: toggle reward plot overlay

Important:
- MJLab `play` does not provide direct keyboard control for velocity command setpoints in this task.
- Commands are generated by the task command manager.

## 6.2 ONNX inference script controls

Script:
- `scripts/infer_policy.py`

This script has explicit keyboard control mappings for command injection.

Velocity mode controls:
- `Up/Down`: linear x command
- `Left/Right`: linear y command
- `A` / `E`: yaw command
- `Space`: zero command

Additional modes:
- `B`: body pose mode toggle
- `H`: head mode toggle
- `G`: ground-pick trigger (if loaded)

Note:
- `scripts/infer_policy.py` is currently MicroDuck-oriented (`MICRODUCK_XML` and sensor/body naming assumptions).
- Use it as a controls reference unless you adapt it for Waddle XML/sensor names.

## 7. `.pt` vs `.onnx` in this repo

### `.pt` (training checkpoints)
- Files like `model_250.pt`, `model_500.pt`.
- Saved during training at `save_interval`.
- Used for:
  - resume training
  - `uv run play ... --checkpoint-file ...`
- Contains full training checkpoint state needed by runner (policy and training state).

### `.onnx` (exported inference model)
- A serialized computation graph + weights for policy inference interoperability.
- In this velocity pipeline, ONNX is exported from runner save path.
- Used for deployment/inference in ONNX runtimes.

### What ONNX additionally stores here
This repo attaches metadata to ONNX files (from MJLab exporter utilities), including:
- run path
- joint names
- joint stiffness and damping
- default joint position
- command names
- observation names
- action scale

Source paths:
- `.venv/lib64/python3.12/site-packages/mjlab/tasks/velocity/rl/runner.py`
- `.venv/lib64/python3.12/site-packages/mjlab/tasks/velocity/rl/exporter.py`
- `.venv/lib64/python3.12/site-packages/mjlab/rl/exporter_utils.py`

## 8. Quick visual evaluation workflow

1. Train short scout run:

```bash
CUDA_VISIBLE_DEVICES=11 uv run train Mjlab-Velocity-Flat-Waddle \
  --env.scene.num-envs 4096 \
  --agent.max-iterations 1000 \
  --agent.save-interval 50
```

2. Test an early checkpoint:

```bash
uv run play Mjlab-Velocity-Flat-Waddle \
  --agent trained \
  --checkpoint-file logs/rsl_rl/waddle_velocity/<run_folder>/model_250.pt \
  --num-envs 1 \
  --viewer native
```

3. If bouncing is too aggressive after `500-1000` iterations, tune:
- command ranges
- `joint_pos_action.scale`
- `action_rate_l2` / `joint_torques_l2` / `foot_slip` penalties
- foot target heights

4. Re-run and compare with one change at a time.
